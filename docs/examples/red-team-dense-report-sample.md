## 1. Verdict

**Not ready to build.** The artifact's headline claim — "we don't see a way for a request to be forged or replayed within the design's assumptions" (`DESIGN.md:35-36`) — is false: there are **two independent replay bypasses, each Critical on its own**, plus a signature-coverage gap that allows silent tampering of authenticated requests. Neither Critical is fixed by the other's mitigation; both structural fixes are required before this ships.

## 3. Findings (Critical → Low)

### R1 — Replay protection is fully bypassable: the nonce isn't signed — **Critical**
- **Scenario.** Attacker captures one valid request (MITM on the internal network, a proxy log, a mirrored packet). They resend the identical body and identical `X-Sig`, but swap in a fresh random `X-Nonce`. `receiver` recomputes `HMAC(secret, method+path+body)` — unchanged, so it verifies — and the new nonce has never been seen, so the replay guard passes. Request accepted. The one control meant to stop replay is trivially defeated.
- **What it targets.** D4 — the decision that the signature covers only `method + path + body` and deliberately excludes `X-Nonce` and any timestamp ("the body already makes each request unique").
- **Evidence.** `[VERIFIED]` `DESIGN.md:22-24` — "The signature does **not** cover the `X-Nonce` header or any timestamp — signing `method + path + body` is enough." `DESIGN.md:18-20` — the nonce is the sole replay guard. The nonce being unauthenticated means the attacker controls it freely.
- **Mitigation.** Put `X-Nonce` **and** a timestamp inside the signed material: `HMAC(secret, method + full-target + body + nonce + timestamp)`. Reject any request whose timestamp is outside a small skew window (this also closes the "replayable forever" gap folded in from F7). Prerequisite for R2's fix to mean anything.

### R2 — Per-replica nonce caches let any replay through the next replica — **Critical**
- **Scenario.** `receiver` runs 3 replicas behind a round-robin LB, each with its **own** in-memory nonce cache and no shared store. Attacker replays a captured request (even a fully-signed one from R1's fix). The LB routes the replay to a different replica, which has never recorded that nonce, and accepts it. On a 3-replica round-robin, a replay succeeds roughly 2 out of 3 times on the first try, ~100% within a few tries.
- **What it targets.** D5 — 3 horizontally-scaled replicas behind round-robin, each keeping an independent nonce cache "to keep the hot path lock-free and fast."
- **Evidence.** `[VERIFIED]` `DESIGN.md:25-27` — "3 horizontally-scaled replicas behind a round-robin load balancer. Each replica keeps its own independent nonce cache (no shared store)."
- **Mitigation.** Replace per-replica caches with a **shared** nonce store (e.g. Redis) with atomic check-and-set, or route deterministically on the nonce so a given nonce always hits the same replica. **Critical adjudication point: this is independent of R1.** Signing the nonce (R1) does nothing here, and sharing the store (R2) does nothing against R1's fresh-nonce replay — you must ship **both** fixes.

### R3 — Query string is unsigned: silent tampering / authorization bypass — **High**
- **Scenario.** Attacker captures a valid signed request to `/api/action?role=user` and rewrites it to `?role=admin` (or `?account=…`, `?amount=…`). Because only `path` — not the query string — is signed, the HMAC still verifies and `receiver` honors the tampered parameters.
- **What it targets.** D1/D4 — the signed material is `method + path + body`; the query string is not part of "path."
- **Evidence.** `[VERIFIED]` `DESIGN.md:10-12` (signature = `method + path + body`) and `DESIGN.md:22-24` (nothing beyond those three fields is signed).
- **Mitigation.** Sign the **full request target** (path + query string), folded into R1's expanded payload — one signed-material change covers both. Severity rises to **Critical** if any query parameter carries privilege, identity, or amount; High as a floor because impact is conditional on what the query encodes.

### R4 — Ephemeral, bounded nonce cache reopens replay windows — **High**
- **Scenario.** The cache is in-memory and holds only the last 100k nonces per replica. Two silent failure modes: (a) any deploy, crash, or restart empties it, so every previously-seen nonce becomes replayable again; (b) under load, once >100k nonces arrive, the LRU evicts the oldest, and those requests become replayable. A recurring, unmonitored replay window that widens exactly when traffic is high.
- **What it targets.** D3/D5 — the in-memory, per-replica, 100k-entry LRU nonce cache.
- **Evidence.** `[VERIFIED]` `DESIGN.md:18-20` — "in-memory LRU cache of the last 100k nonces."
- **Mitigation.** Use a durable, shared store with a **TTL tied to the signed timestamp** from R1 (entries only need to live as long as the accepted clock-skew window, which also bounds storage). Same store as R2 — see de-fang note.

### R5 — Single long-lived secret with no rotation path — **High**
- **Scenario.** One shared secret, "set once, trusted forever," stored in each service's environment. If it leaks — via a process listing, a crash dump, a CI log, an errant `env` dump — there is no key-id and no rollover: revoking means redeploying every caller and receiver simultaneously, incurring an outage, with no overlap window. Recovery is an incident, not a rotation.
- **What it targets.** D2 — a single long-lived secret with rotation explicitly deferred.
- **Evidence.** `[VERIFIED]` `DESIGN.md:14-16` — "single long-lived value… set once, trusted forever." (F12's "secret could be exposed via process listing / crash dumps / CI logs" is the same root cause — no additional concrete exposure path exists in the artifact, so it's subsumed here, not a separate finding.)
- **Mitigation.** Add a `key-id` header and support overlapping-validity rotation (accept old + new during a window). This need not block the first build, but the header and multi-key lookup should be designed in now — retrofitting a key-id into a signed protocol later is itself a breaking change.

### R6 — Full-request error logging: secret/PII leakage + attacker-controlled log-flood DoS — **High**
- **Scenario.** On any verification failure, `receiver` logs the full request — headers and body — at ERROR, then returns 401. Two bites: (a) attacker-supplied bodies (tokens, PII, log-injection payloads) and headers land verbatim in logs, and anyone with log access now has them; (b) the failure path is the cheapest request an attacker can send, and each one triggers an expensive full-body ERROR log — a volume-amplified log-storage/-cost DoS the attacker controls directly. (F11 — flooding to inflate log cost — is the same mechanism, folded here.)
- **What it targets.** D6 — log full request (headers + body) at ERROR on signature-verification error.
- **Evidence.** `[VERIFIED]` `DESIGN.md:29-30` — "logs the full request (headers + body) at ERROR level… then returns 401."
- **Mitigation.** Log a **fingerprint** (hash of method+target, nonce prefix, a request id) — never the raw body or headers — and rate-limit / sample failure logs so volume can't be driven by an attacker.

### R7 — Duplicate processing of legitimate retries — **Medium**
- **Scenario.** A benign client retries after a 5xx and reuses the same nonce. The origin replica rejects it as a repeat, but round-robin sends the retry to a second replica that never saw the nonce, which accepts it → the non-idempotent work runs twice (double charge, double write).
- **What it targets.** D5 — per-replica independent caches (a correctness consequence of the same structure R2 attacks for security).
- **Evidence.** `[VERIFIED]` `DESIGN.md:25-27`.
- **Mitigation.** The shared nonce store from R2 removes the cross-replica gap; additionally make the affected handlers idempotent so a legitimately-retried nonce is safe regardless. Largely de-fanged by R2 (see notes).

### R8 — Comparison not specified constant-time — **Low** (demoted from Medium)
- **Scenario.** If `receiver`'s HMAC comparison uses a byte-early-exit `==`, response timing leaks how many leading bytes matched, theoretically enabling forgery over a very large number of probes.
- **What it targets.** D1 — "`receiver` recomputes and compares."
- **Evidence.** `[UNCONFIRMED]` `DESIGN.md:10-12` says "compares" but does not specify constant-time; no implementation exists to inspect. **Adjudication:** demoted to Low — the finding is a design *omission*, not a demonstrated flaw, and network-jitter-masked timing attacks against HMAC comparison are notoriously hard to exploit in practice. The fix is nearly free, so it's worth stating.
- **Mitigation.** State in the design that comparison MUST be constant-time (`hmac.compare_digest` or equivalent).

### R9 — Missing-nonce failure is indistinguishable from a bad signature — **Low**
- **Scenario.** A legitimate client with a bug that omits `X-Nonce` gets a bare 401, identical to a signature failure, making the misconfiguration slow to diagnose.
- **What it targets.** D3 — "No nonce → rejected."
- **Evidence.** `[VERIFIED]` `DESIGN.md:18-20`.
- **Mitigation.** Return a distinct, non-sensitive error code for "missing nonce" vs "bad signature." (Keep the codes coarse enough not to become an oracle — distinguish client-shape errors, not cryptographic verdicts.)

## 4. Adjudication notes (where I didn't rubber-stamp)

- **The two Criticals are genuinely independent — do not let one fix create false confidence.** F2 already flagged, and I confirm, that signing the nonce (R1) does **not** help against the per-replica-cache bypass (R2), and a shared store (R2) does **not** help against the fresh-nonce replay (R1). Shipping either fix alone leaves a complete replay bypass standing. This is the single most important takeaway and the reason the "why we're confident" paragraph is wrong.
- **De-fang collapse — one store resolves three findings.** R2, R4, and the security half of R7 all stem from the per-replica in-memory LRU. A **single mitigation** — a shared, durable nonce store with a TTL bound to the signed timestamp — dissolves all three. R7 additionally wants idempotent handlers as defense-in-depth.
- **De-fang collapse — one signed-payload change resolves two findings.** R1 and R3 (and the folded F7 "replayable forever") are all signature-*coverage* gaps. Expanding the signed material once to `method + full-target(path+query) + body + nonce + timestamp` closes all of them.
- **F6 demoted, not dropped.** It's `[UNCONFIRMED]` and describes an omission rather than an observed flaw; timing side channels here are low-exploitability. Kept as Low because the hardening is cheap and belongs in the written design.
- **F12 refuted as standalone — upheld.** The verifier's refutation is correct: it adds no concrete exposure path beyond R5's "no rotation" root cause. Folded into R5.
- **No new severe finding surfaced in adjudication.** I checked whether full-request logging (D6) could leak the shared *secret* itself — it can't directly, since the secret lives in env and never appears in the request; R6 already captures the token/PII exposure that does occur.
- **Line pins normalized.** The verifier's citations were offset by ~2 lines in places (e.g. it cited D3 at `:16-18`); I re-pinned every reference to the artifact as actually read.

## 6. Shared-bias + cost footer

All reviewers were Claude (Opus 4.8); **model-mix off** — adversary, verifier, and this adjudicator share one model family, so a blind spot common to that family (e.g. an unconventional-but-valid crypto construction misread as flawed, or a shared miss) would survive all three stages. Treat the crypto-coverage findings as high-confidence and the operational ones (R7, R9) as worth a human skim. Approx **3 model calls** (1 adversary + 1 verifier + 1 adjudication).

## 7. Do this first (dependency-ordered)

1. **Expand the signed payload** to `HMAC(secret, method + full-target(path+query) + body + nonce + timestamp)`, and reject requests outside a small timestamp-skew window. *(Fixes R1, R3; closes forever-replay. Do this first — it defines the record the nonce store keys on.)*
2. **Replace the per-replica in-memory LRU with a shared, durable nonce store**, with atomic check-and-set and a TTL equal to the skew window from step 1. *(Fixes R2, R4, and the security half of R7 in one move. Depends on step 1 for the timestamp/TTL.)*
3. **Make non-idempotent handlers idempotent** as defense-in-depth against duplicate retries. *(Completes R7.)*
4. **Stop logging raw requests on failure** — log a fingerprint only, and rate-limit/sample failure logs. *(Fixes R6; independent, can be done in parallel.)*
5. **Design in a `key-id` header + overlapping-validity rotation** now, even if rotation itself ships later. *(Addresses R5; retrofitting a key-id into a signed protocol is a breaking change, so reserve the header before first release.)*
6. **Specify constant-time comparison** in the design and use `hmac.compare_digest` or equivalent. *(Fixes R8; cheap.)*
7. **Return a distinct, coarse error for missing-nonce vs bad-signature** — without turning it into a cryptographic oracle. *(Fixes R9.)*

**Gate:** steps 1 and 2 are both mandatory before build — either alone leaves a complete replay bypass open, so the artifact's converged "cannot be forged or replayed" claim does not hold until both land.
