## 1. Frame (restated)
- **Question:** Background-job queue for the new notifications service; ~500 jobs/sec
  bursty (email/push, retries, scheduled sends).
- **Options:** A = Postgres `SKIP LOCKED`; B = Redis + BullMQ; C = managed cloud queue.
- **Confirmed principle (stated):** "prefer the smallest operational surface area — reuse
  infrastructure we already run and understand over adopting new systems, unless the new
  system is clearly necessary." (Subordinates raw-throughput-max and feature-richness.)
- **Constraints (classification):** G1 self-hostable/in-VPC for compliance — A: pass;
  B: pass; C: **fail**.
- **Stakes/reversibility:** Moderately reversible (producer/consumer API can be
  abstracted); a wrong throughput call surfaces only under production burst.
- **Gate-dropped:** C (payloads leave the environment — violates the hard constraint).

## 2. Constraint check
A — PASS G1. B — PASS G1. C — **FAIL** G1 → eliminated before ranking. Contest is A vs B.

## 3. Recommendation — by principle
**Choose A (Postgres `SELECT … FOR UPDATE SKIP LOCKED`), because it best serves "prefer
the smallest operational surface area … unless the new system is clearly necessary."**
The only fact that could make B "clearly necessary" — that 500 jobs/sec exceeds
Postgres's reach — is refuted by verified evidence. Does **not** override the advocate
majority: advocate B conceded the principle favors reuse and that its durability is
weaker, resting its case solely on "clearly necessary," which the evidence defeats.

## 4. Crux resolution (decomposed → verified)
- **CRUX-1 (decisive) — Postgres sustains ~500 jobs/sec.**
  - 1a `[VERIFIED]`: *"a well-specced database server can queue around 172,000 jobs
    per second from a single PostgreSQL client … process around 196k jobs per second"* —
    worker.graphile.org/docs/performance. 500/sec ≈ 0.3% of this. **Demoting caveat
    (quoted):** *"optimial conditions"*, on one dev box running DB+workers together — not
    a production app-load-sharing config.
  - 1b `[VERIFIED]`: *"the ceiling is lower than most people expect"* but *"Under 100
    concurrent workers, simple jobs – Postgres with SKIP LOCKED is fine"* —
    richyen.com/postgres/2026/05/04/postgres_job_queue.html. 500 short sends/sec fits.
  - **Verdict:** achievable (not `[UNCONFIRMED]`); residual is shared-instance contention
    → CRUX-2. Value residual: none.
- **CRUX-2 — bloat real, and mitigable.**
  - 2a `[VERIFIED]`: *"Dead tuples accumulate faster than autovacuum can clean them"*;
    *"dead tuples grow about 14×, … table size about 15×, and dequeue throughput drops
    about 35%"* — github.com/NikolayS/pgque.
  - 2b `[VERIFIED]`: *"use TRUNCATE or drop partitions … vacuuming is not needed and
    bloat is not an issue"*; *"TRUNCATE rotation creates zero dead tuples by
    construction"* — github.com/NikolayS/pgque.
  - **Value residual → principle:** the partition/TRUNCATE work on familiar, already-
    backed-up infra is a smaller operational surface than standing up + securing a new
    Redis → principle selects A.
- **CRUX-3 — Redis/BullMQ durability weaker (B's own concession, verified).**
  - 3a `[VERIFIED]`: *"Many hosting solutions do not offer persistence by default …
    needs to be configured per instance."* — docs.bullmq.io/guide/going-to-production.
  - 3b `[VERIFIED]`: *"We recommend enabling AOF …"*, *"1 second per write is enough
    for most applications"* (~1s loss window), *"maxmemory-policy … noeviction … the only
    setting that guarantees correct behavior."* — same source.
  - **Verdict:** confirmed; B is a strict durability regression vs A's free WAL durability.

## 5. Concession trace
- **R1 openings:** A — HOLD(op-surface), HOLD(transactional), NEW-CRUX(throughput).
  B — **CONCEDE**(durability config-dependent/weaker), HOLD(scheduling), NEW-CRUX(shared-DB
  degradation), COUNTER(throughput is shared).
- **R1 rebuttals:** A — COUNTER(bloat mitigable via partition/TRUNCATE), **CONCEDE**(failure
  couples to primary DB → crux must be verified), NEW-CRUX(is 500/sec in documented range).
  B — **CONCEDE**(bloat not inherent given mitigation), COUNTER(mitigation is new complexity).
- **R2:** A — **CONCEDE**("zero new surface" → "smaller new surface; tuning is new work").
  B — HOLD all, nothing new.
- **R3:** both HOLD, nothing new → **mechanical stop** (no new CONCEDE, no new NEW-CRUX).
- Every position accounted for; C dropped at the gate.

## 6. What you lose
- **Strongest case for B (road not taken):** purpose-built — blocking pops (no polling),
  native scheduling/retry/rate-limiting, and **physical isolation of queue churn from the
  primary app DB**, removing the "queue load degrades the product's main DB" risk class.
- **Frontrunner's strongest unrebutted weakness:** no verified source measured Postgres
  SKIP-LOCKED churn *sharing an instance with concurrent production OLTP*; the 172k/sec
  headroom was *"optimial conditions"* on a dedicated dev box. The verdict extrapolates
  ample shared-instance headroom — reasonable, but not measured for this mixed workload.

## 7. Confidence + revisit trigger
- **Moderate-to-high** on option A, **conditional** on the shared-instance assumption
  (throughput feasibility, bloat mitigability, B's durability regression all `[VERIFIED]`;
  shared-instance behavior extrapolated).
- **Rests explicitly on the assumption** (one-way-door logic): Postgres can absorb ~500
  jobs/sec churn without degrading primary OLTP, given a partitioned/TRUNCATE-rotated
  table + tuned autovacuum.
- **Hard revisit triggers → switch to in-VPC Redis if:** (a) load-test at 1.5× peak shows
  primary-app p99 regression beyond SLO; (b) bloat keeps climbing despite TRUNCATE
  rotation; (c) workload outgrows ~100 workers or needs heavy scheduling/rate-limiting.
  Validate (a)/(b) **before** production traffic.

## 8. Shared-bias + cost footer
- **Shared-bias:** both this panel and the team's principle lean "reuse what we run" —
  real risk of mutually reinforcing status-quo bias; the shared-instance risk (slot 6) is
  where it's most likely hiding. Mitigated by requiring B be refuted on the merits
  (throughput verified, not assumed), and by gating on the load test.
