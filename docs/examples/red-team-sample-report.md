## Verdict
**NOT ready to build.** 8 adversary findings → **6 kept** (after folding): **1 Critical,
2 High, 2 Medium, 1 Low**. The headline "we don't see a way for this to leak secrets or
grant unintended capability" is **refuted** — both a secret-leak path and an
arbitrary-capability grant are verified.

## Severity-ranked findings

**[CRITICAL] F2/F5 — `Bash(python *)` is arbitrary code execution, not "Python-scoped"**
- Scenario: The unattended background agent (or any prompt-injected content it reads)
  runs `python -c "import os,socket,subprocess;..."` — a reverse shell, secret
  exfiltration, or `rm -rf` — auto-approved because it matches the allow-rule. No human
  is watching ("run unattended", DESIGN.md:3-4).
- Evidence **[VERIFIED]**: `.claude/settings.local.json:4` `"Bash(python *)"`; arbitrary
  execution is intrinsic to `python -c`. Anthropic's own example settings treat a ` *`
  wildcard command (`"Bash(curl *)"`) as deny-list material (code.claude.com/docs/en/settings).
- Mitigation: Do not grant `Bash(python *)`. Grant the exact invocation only —
  `"Bash(python -m pytest)"` (+ `"Bash(python -m pytest *)"` if args needed). Prereq:
  first fix F3 (use the binary that actually exists). Consider sandboxing the agent.

**[HIGH] F1 — "It's gitignored, so it's private" rests on a machine-local global config, not the repo**
- Scenario: A second developer creates `.claude/settings.local.json` by hand (the design
  tells them to store local paths there). Their global git has no such ignore. They
  `git add .` and commit it. The "private" file — with absolute home paths and broad
  grants — lands in the shared repo.
- Evidence **[VERIFIED/REFUTED-as-stated]**: project `.gitignore:1-5` does not list it;
  it is ignored only via `~/.config/git/ignore:1` (the developer's global excludes,
  not the repo). Docs: the
  auto-ignore guarantee holds only when Claude Code creates the file
  (code.claude.com/docs/en/settings).
- Mitigation: Add `.claude/settings.local.json` to the project `.gitignore` explicitly.
  Don't rely on per-developer global config or on Claude having created the file.

**[HIGH] F3/F8 — Bootstrap validates `python3` but the design grants/runs `python`; false-green**
- Scenario: `bootstrap.sh` prints "Bootstrap complete," the agent runs `python -m
  pytest`, and fails with `command not found` — this machine has only `python3`. The
  unattended run dies after reporting healthy.
- Evidence **[VERIFIED]**: `bootstrap.sh:6` checks `python3`; DESIGN.md:14 and
  `settings.local.json:4` use `python`; `command -v python` returns nothing on this host.
- Mitigation: Pick one binary end-to-end (recommend `python3`): grant
  `Bash(python3 -m pytest)`, run `python3 -m pytest`, keep the `python3` check. Settle
  this before F2's rule wording, since the rule string depends on the chosen binary.

**[MEDIUM] F4 — "Tightly scoped" is false; undocumented rules already present**
- Scenario: Reviewer trusts "tightly scoped" and never audits the file; `Bash(npm test)`
  and `Read(./src/**)` (absent from the design) silently widen the agent's reach.
- Evidence **[VERIFIED]**: `settings.local.json:5-6`.
- Mitigation: Reconcile the file to the design — remove rules with no stated purpose, or
  document why each exists.

**[MEDIUM] F7 — "Never prune" + unreviewed private file = permission rot**
- Scenario: Over months, removed tooling leaves broad grants behind on a file that never
  appears in PR review; the agent's privilege only grows.
- Evidence **[VERIFIED]**: Decision 3 (DESIGN.md:17-19) + F4's already-orphaned rules.
- Mitigation: Drop "no pruning needed." Prune on tool removal; periodically diff the
  allow-list against actually-used tools.

**[LOW] F6 — Hardcoded user paths in env**
- Scenario: If F1 fails and the file is committed, `/Users/jdoe/...` leaks a username and
  internal-tooling layout.
- Evidence **[VERIFIED]**: `settings.local.json:10-11`.
- Mitigation: Subsumed by fixing F1 (keep it out of the repo). If paths must be shared,
  parameterize via env interpolation rather than hardcoding.

## Adjudication notes (where the orchestrator did not rubber-stamp)
- **F1 — pushed back on the adversary AND the design.** The adversary said "not
  gitignored"; that's wrong — `git check-ignore` proves it IS ignored. But the design's
  *reasoning* is still wrong: it's ignored by the developer's global config, not the
  repo, so the privacy property doesn't travel to teammates. Kept F1 as High, reframed
  from "false" to "true by luck, fragile by design." Verify-don't-assert cut both ways.
- **F5 internal-matcher claim demoted to [UNCONFIRMED] but finding kept.** Couldn't pin
  from docs whether `&&`/`;` chaining defeats the matcher; didn't let the finding rest on
  that — `python -c` alone is arbitrary execution, verified independently.
- **De-fang check:** Fixing **F2** (exact-invocation rule) dissolves **F5** (same grant —
  folded) and largely neuters **F6**. Fixing **F1** (explicit project `.gitignore` entry)
  dissolves the leak trigger for **F6** and the committed-file half of **F7**.
